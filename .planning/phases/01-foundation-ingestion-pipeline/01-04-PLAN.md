    **frontend/src/api/types.ts** — Add upload-related types:
    - `UploadResponse`: { batch_id: number, task_id: string, filename: string, message: string }
    - `BatchResponse`: { id: number, data_source_id: number, filename: string, uploaded_by: string, row_count: number | null, status: string, error_message: string | null, created_at: string, task_id: string | null }
    - `TaskStatus`: { task_id: string, state: string, stage: string | null, progress: number | null, detail: string | null, row_count: number | null }
    - `ColumnDetectResponse`: { columns: string[] }

    **frontend/src/hooks/useTaskStatus.ts** — Task polling hook:
    - `useTaskStatus(taskId: string | null)`: Uses TanStack Query with `refetchInterval: 1000` (poll every second) when taskId is not null and state is not 'COMPLETE' and not 'FAILURE'. Returns { state, stage, progress, detail, row_count, isComplete, isFailed }.
    - Stop polling when state is COMPLETE or FAILURE.

    **frontend/src/components/DropZone.tsx** — Drag-and-drop upload zone (USE frontend-design SKILL):
    - Large dashed-border drop zone area
    - "Drag & drop your CSV file here" text with icon
    - "Browse files" button inside the zone (both drag-drop and click-to-browse work)
    - Accept only .csv files
    - Visual feedback on drag-over (border color change, background tint)
    - Props: `onFileSelected(file: File)`, `disabled: boolean`
    - Shows selected filename before upload starts

    **frontend/src/components/ColumnMapper.tsx** — Visual column mapper (USE frontend-design SKILL):
    - Per CONTEXT.md locked decision: "left column shows canonical fields, right column shows dropdowns populated with actual CSV headers from the uploaded file"
    - Layout: Two-column grid. Left: canonical field label (supplier_name*, supplier_code*, short_name, currency, payment_terms, contact_name, supplier_type). Right: `<select>` dropdown populated with detected CSV column headers + an empty "—" option.
    - Required fields (supplier_name, supplier_code) marked with asterisk and validation
    - Submit button to create the source with the mapping
    - Props: `columns: string[]` (detected CSV headers), `onSubmit(sourceData: DataSourceCreate)`, includes name and description inputs for the new source
    - Clear visual hierarchy: "Step 1: Name your source" → "Step 2: Map columns" → "Create & Upload"

    **frontend/src/components/ProgressTracker.tsx** — Pipeline progress display (USE frontend-design SKILL):
    - Per CONTEXT.md locked decision: "the upload area transforms into a progress tracker"
    - Shows 4 stages as a horizontal pipeline: Parsing → Normalizing → Embedding → Matching Enqueued
    - Each stage shows: icon, label, status (pending/active/complete)
    - Active stage has animation (spinner/pulse) — per Claude's discretion for loading animations
    - Progress percentage shown for active stage
    - On completion: summary line — "1,623 rows processed. 12 warnings." with expandable details
    - On failure: error message display with red styling
    - Props: `taskId: string`, uses useTaskStatus hook internally

    **frontend/src/components/ReUploadDialog.tsx** — Re-upload confirmation (USE frontend-design SKILL):
    - Per CONTEXT.md locked decision: Shows impact counts — "EOT already has 1,623 staged suppliers from batch #3. Uploading will supersede those records and invalidate 42 pending match candidates. Continue?"
    - Modal dialog with dark overlay
    - Shows source name, existing record count, pending match count
    - "Cancel" and "Continue Upload" buttons
    - Props: `sourceName: string`, `existingCount: number`, `pendingMatchCount: number`, `onConfirm()`, `onCancel()`
    - NOTE: For the counts, query GET /api/import/batches?data_source_id={id} to check if previous batches exist. If the backend doesn't have a dedicated re-upload-info endpoint, show a simpler message: "{Source} has previous uploads. New upload will supersede existing staged records. Continue?" This can be enhanced when Phase 2 adds match candidates.

    **frontend/src/components/BatchHistory.tsx** — Batch history table (USE frontend-design SKILL):
    - Per CONTEXT.md locked decision: "Batch history visible under data source — user can see all previous uploads, row counts, timestamps, and superseded status"
    - Table showing: filename, uploaded_by, row_count, status (with color coding: completed=green, failed=red, processing=yellow, superseded=gray), created_at
    - Sorted by created_at descending (newest first)
    - Props: `dataSourceId: number` — fetches batches via GET /api/import/batches?data_source_id={id}
    - Use TanStack Query for data fetching

    **frontend/src/pages/Upload.tsx** — Upload page orchestrator (USE frontend-design SKILL):
    - **State machine with 4 states:**
      1. `SELECT_SOURCE`: Dropdown to select existing source OR "New source" option. Shows batch history for selected source.
      2. `UPLOAD_FILE`: DropZone for file selection. If existing source selected, show file drop zone directly. If "New source" chosen, proceed to mapping after file selection.
      3. `MAP_COLUMNS` (new source only): After file selected, detect columns via API, show ColumnMapper. On submit, create source then trigger upload.
      4. `PROCESSING`: DropZone transforms into ProgressTracker. Show real-time progress. On completion show summary.
    - Re-upload flow: When existing source is selected and file is dropped, check if source has existing batches. If yes, show ReUploadDialog before proceeding.
    - After processing completes, show "Upload another file" button to reset state
    - Source selector dropdown at top with all available sources from GET /api/sources
    - Batch history section below the upload area (always visible when a source is selected)
